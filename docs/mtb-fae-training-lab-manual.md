## Lab 1: Clone, Resolve Dependencies & Build

**Segment:** 1 — Working Effectively with Version Control & Git
**Tools:** Modus-Shell (command line only)

> 💡 **Lab 1 uses Modus-Shell exclusively.** All subsequent labs are done in the Eclipse IDE (or VS Code if that is your preference — lab steps document Eclipse).

### Objective

Understand the Git + `make getlibs` workflow — clone a real ModusToolbox project from GitHub, resolve its dependencies, and build it. This is the foundational workflow for every ModusToolbox project you'll encounter in the field.

### Background

ModusToolbox projects use a lightweight dependency system built on `.mtb` files — small text files that contain a Git URL and version tag pointing to a library. Here's what you need to know:

| Concept                     | What It Means                                                                                                                                                                    |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`.mtb` file**             | A text file containing a Git URL + version tag pointing to a library.                                                                                                            |
| **`deps/` directory**       | YOUR declared dependencies — the `.mtb` files you (or Project Creator / Library Manager) put here. **Committed to Git.**                                                         |
| **`bsps/` directory**       | YOUR BSP selection — the `.mtb` file for your target board, plus local BSP configuration files. **Committed to Git.**                                                            |
| **`libs/` directory**       | Generated by `make getlibs`. Contains the resolved transitive `.mtb` files (dependencies of your dependencies) AND the generated `mtb.mk` makefile that wires everything into the build. **Never committed to Git.** |
| **`mtb_shared/` directory** | The actual cloned library source code. Lives ONE LEVEL ABOVE your project and is shared across all projects in the workspace. **Never committed to Git.**                         |
| **`make getlibs`**          | The dependency resolver. Reads all `.mtb` files in `deps/` and `bsps/`, resolves the full dependency tree, clones source code into `mtb_shared/`, and generates the build wiring in `libs/`. |

`deps/` and `bsps/` contain your declared dependencies — the `.mtb` files you control. `make getlibs` resolves the full dependency tree, generates the build wiring in `libs/`, and clones the actual source code into `mtb_shared/`. You commit your declarations; everything else is reconstructed.

### Step-by-Step Instructions

#### Step 1 — Open Modus-Shell

Launch Modus-Shell from your Windows Start Menu. You should see a terminal prompt with the ModusToolbox environment configured.

#### Step 2 — Create the training workspace

```bash
mkdir C:/MTB_Training && cd C:/MTB_Training
```

**Expected result:** You are now in an empty `C:/MTB_Training/` directory.

#### Step 3 — Clone the training project

```bash
git clone https://github.com/ClarkJ-Infineon/mtb-training-blinky
```

**Expected result:** Git clones the repository. You should see output like:

```
Cloning into 'mtb-training-blinky'...
remote: Enumerating objects: ...
Receiving objects: 100% ...
```

> ⚠️ **Notice:** There is no `mtb_shared/` or `libs/` directory yet — only the source code and dependency declarations were committed to Git.

#### Step 4 — Enter the project directory

```bash
cd mtb-training-blinky
```

#### Step 5 — Inspect the declared dependencies

```bash
ls deps/
```

**Expected result:** You should see the project's direct dependency `.mtb` files (e.g., `retarget-io.mtb`). Each file is a simple text pointer to a Git repository and version. Note that the BSP is in `bsps/`, not `deps/`.

To see what's inside an `.mtb` file:

```bash
cat deps/retarget-io.mtb
```

**Expected result:** A line containing a URL, version tag, and local path — for example:

```
https://github.com/cypresssemiconductorco/retarget-io#release-v1.9.0#$$ASSET_REPO$$/retarget-io/release-v1.9.0
```

#### Step 6 — Resolve dependencies with `make getlibs`

```bash
make getlibs
```

**Expected result:** The tool reads all `.mtb` files in `deps/`, clones each library into `../mtb_shared/`, and generates `libs/mtb.mk`. You'll see Git clone progress for each dependency. This may take 1–3 minutes depending on your internet connection.

When complete, you should see output ending with something like:

```
Library Manager: Completed
```

#### Step 7 — Inspect the `libs/` directory

```bash
ls libs/
```

**Expected result:** You should see many more `.mtb` files than were in `deps/` — these are the **transitive dependencies** (dependencies of your dependencies). You'll also see `mtb.mk` and other generated `.mk` files.

#### Step 8 — Inspect `libs/mtb.mk`

```bash
cat libs/mtb.mk
```

**Expected result:** A series of `SEARCH_*` variable definitions, each pointing to a library path under `../mtb_shared/`. These paths are how the build system finds library source code and headers. For example:

```makefile
SEARCH_retarget-io=../mtb_shared/retarget-io/release-v1.9.0
SEARCH_mtb-hal-psc3=../mtb_shared/mtb-hal-psc3/release-v1.1.2
...
```

#### Step 9 — Build the project

```bash
make build
```

**Expected result:** The GCC ARM compiler builds the project targeting `KIT_PSC3M5_EVK`. Build output scrolls by, and the final lines should indicate success:

```
==============================================================================
= Build complete =
==============================================================================
```

The build output (`.elf`, `.hex` files) is placed in the `build/` directory.

## Lab 2: Import into Eclipse and Export with mtbarchive

**Segment:** 2 — Sharing, Importing & Exporting Projects
**Tools:** ModusToolbox Eclipse IDE

### Objective

Import the cloned project into the Eclipse IDE and create a smart archive using `mtbarchive`. This demonstrates the two most common project-sharing workflows: IDE collaboration and archive-based handoff.

### Background

**IDE Import:** ModusToolbox projects are IDE-agnostic — they're built with Makefiles, not IDE project files. To work in Eclipse, you "import in-place," which registers the existing project folder with Eclipse without copying or restructuring anything.

**mtbarchive:** A ModusToolbox-aware archiving tool that creates self-contained project ZIPs. Unlike a plain ZIP, `mtbarchive` knows which files are authored (include them) and which are generated (exclude them). The result is a small, clean archive that any recipient can restore with `make getlibs` → `make build`.

| ✅ Included by mtbarchive | ❌ Excluded by mtbarchive |
|---|---|
| Source code, Makefile, configs | `libs/` (regenerated by `make getlibs`) |
| `deps/` (`.mtb` dependency declarations) | `build/` output |
| `bsps/` (BSP configuration & device config) | `mtb_shared/` (regenerated) |
| `.git/` history (if present) | IDE workspace files |
| README, LICENSE | |

## Lab 3: Create a Custom BSP

**Segment:** 4 — Workflows to Develop & Utilize a Custom BSP
**Tools:** Eclipse IDE, Library Manager, Device Configurator

### Objective

Create a custom Board Support Package (BSP) from the kit BSP, modify pin assignments, and build with the new BSP. This is the exact workflow you'll guide customers through when they move from an eval board to their production hardware.

### Background

**Why custom BSPs matter:** Infineon kit BSPs are a starting point — they provide a working configuration for the eval board. But most production designs have different pin assignments, crystal frequencies, debug interfaces, and memory layouts. Creating a dedicated custom BSP gives the customer a clean baseline for their specific hardware.

**The workflow:**

| Step | Action |
|---|---|
| 1 | Copy the closest kit BSP |
| 2 | Rename to `TARGET_<your_name>` |
| 3 | Set it as the active BSP (via Library Manager) |
| 4 | Edit `design.modus` for your hardware (Device Configurator) |
| 5 | Build, test, iterate |

**Key BSP files:**

| File | Purpose | Tool |
|---|---|---|
| `config/design.modus` | Pin assignments, clock tree, peripheral config | Device Configurator |
| `config/GeneratedSource/` | Auto-generated C code from `design.modus` | Device Configurator (auto) |
| `cybsp.h` | Board-level macros (`CYBSP_USER_LED`, `CYBSP_DEBUG_UART`, etc.) | Auto-generated by Device Configurator |
| `cybsp.c` | Board initialization implementation | Text editor (rarely modified) |
| `linker/*.ld` | Memory regions | Text editor (advanced) |

> ⚠️ **Important:** Do not manually edit files in `GeneratedSource/` — they are regenerated from `design.modus` every time you save in Device Configurator. Your edits would be overwritten.

### Prerequisites

- **Lab 1 completed** — project cloned and built in `C:/MTB_Training/mtb-training-blinky`

### Step-by-Step Instructions

#### Step 1 — Copy the kit BSP to create a new BSP

In **Eclipse Project Explorer**, expand `bsps/` in your project. Right-click `TARGET_APP_KIT_PSC3M5_EVK` → **Copy**, then right-click `bsps/` → **Paste**. Rename the pasted folder to `TARGET_MY_BOARD`.

**Expected result:** A new directory `bsps/TARGET_MY_BOARD/` appears in Project Explorer, containing an exact copy of the kit BSP — `config/design.modus`, `cybsp.h`, `cybsp.c`, `GeneratedSource/`, linker scripts, and all other BSP files.

> 💡 **CLI alternative (IDE integrated terminal):** You can also do this from the terminal within Eclipse or VS Code (see Appendix A for terminal setup):
>
> `cd C:/MTB_Training/mtb-training-blinky`
> `cp -r bsps/TARGET_APP_KIT_PSC3M5_EVK bsps/TARGET_MY_BOARD`

> 💡 **Naming convention:** Custom BSP directories always start with `TARGET_`. Use a descriptive name — e.g., `TARGET_MY_PRODUCT_V2`, `TARGET_CUSTOMER_MAIN_BOARD`.

#### Step 2 — Set the new BSP as active using Library Manager

Open **Library Manager** from the Eclipse **Quick Panel** → **Library Manager**.

> 💡 **CLI shortcut (IDE terminal):** You can also launch Library Manager from the IDE's integrated terminal by running `make modlibs`.

In Library Manager:

1. Find `MY_BOARD` in the BSP list
2. Select it as the **active BSP**
3. Click **Update**

**Expected result:** Library Manager updates the Makefile `TARGET=` variable to `MY_BOARD`, regenerates `libs/mtb.mk`, and resolves dependencies for the new BSP.

---

**Two ways to switch the active BSP:**

**Option A — Library Manager GUI (recommended)**

Select the BSP → click **Update**. Library Manager handles everything in one step.

**Option B — Makefile edit + make getlibs (CLI alternative)**

1. Open `Makefile` and change `TARGET=APP_KIT_PSC3M5_EVK` to `TARGET=MY_BOARD`
2. Run `make getlibs` to regenerate `libs/mtb.mk` with the new BSP path

---

#### Step 3 — Open Device Configurator on the new BSP

From Eclipse, open Device Configurator on the `MY_BOARD` BSP:

- Navigate to `bsps/TARGET_MY_BOARD/config/design.modus`
- Double-click to open in Device Configurator

Alternatively, from the Quick Panel → **Device Configurator** (it should target the active BSP).

**Expected result:** Device Configurator opens showing the inherited kit configuration — all the pin assignments, clock settings, and peripheral configurations from the original `KIT_PSC3M5_EVK` BSP.

#### Step 4 — Change one pin assignment

In Device Configurator, modify a pin assignment — for example, move `USER_LED` to a different GPIO pin:

1. Find the current `USER_LED` pin assignment in the Pins tab
2. Disable or reassign it
3. Assign `USER_LED` to a different available GPIO pin

**Expected result:** The pin assignment updates visually in Device Configurator. The old pin is freed, and the new pin shows the `USER_LED` assignment.

#### Step 5 — Save the configurator changes

Click **Save** in Device Configurator (or press `Ctrl+S`).

**Expected result:** Device Configurator regenerates all files in `config/GeneratedSource/`:

- `cycfg_pins.h` / `cycfg_pins.c` — updated pin mux configuration
- `cycfg_clocks.h` / `cycfg_clocks.c` — clock tree configuration
- `cycfg_peripherals.h` / `cycfg_peripherals.c` — peripheral configuration
- `cycfg_system.h` / `cycfg_system.c` — system/power configuration

You can verify the pin change by opening `config/GeneratedSource/cycfg_pins.c` and looking for the new pin assignment.

#### Step 6 — Build with the custom BSP

```bash
make build
```

**Expected result:** The project builds successfully with your custom `MY_BOARD` BSP. The build output should show `TARGET=MY_BOARD` and complete with zero errors.

```
==============================================================================
= Build complete =
==============================================================================
```

#### Step 7 — (Optional) Program and verify

If you have a `KIT_PSC3M5_EVK` eval board connected:

```bash
make program
```

**Expected result:** The firmware is flashed to the board, and the LED blinks on the new pin (if you wired an LED there, or if the new pin has an on-board LED connected).

## Appendix B: Troubleshooting

### `make getlibs` Fails

| Symptom | Cause | Fix |
|---|---|---|
| `fatal: unable to access` or timeout errors | No internet / corporate proxy blocking GitHub | Check network connection; configure Git proxy: `git config --global http.proxy http://proxy:port` |
| `Permission denied (publickey)` | SSH key not configured for GitHub | Use HTTPS URLs in `.mtb` files (default), or configure SSH keys |
| `make: *** No rule to make target 'getlibs'` | Not in a ModusToolbox project directory, or Modus-Shell not configured | Verify you're in the project root (where `Makefile` is); restart Modus-Shell |

### Build Fails After Switching BSPs

| Symptom | Cause | Fix |
|---|---|---|
| Missing header files (`No such file or directory`) | `libs/mtb.mk` is stale — still pointing to old BSP's paths | Run `make getlibs` to regenerate, or use Library Manager to switch BSPs |
| `TARGET` mismatch errors | Makefile `TARGET=` doesn't match BSP directory name | Use Library Manager (not manual edit) to switch BSPs |
| Linker errors referencing wrong memory regions | Stale build artifacts from previous BSP | Run `make clean` then `make build` |

### Missing Headers After Clone

| Symptom | Cause | Fix |
|---|---|---|
| `fatal error: cybsp.h: No such file or directory` | `make getlibs` was not run after cloning | Run `make getlibs` then `make build` |
| `fatal error: cy_pdl_config.h: No such file or directory` | Same — dependencies not resolved | Run `make getlibs` then `make build` |
| Build works on one machine but not another | Different library versions or missing `make getlibs` | Ensure both machines run `make getlibs`; pin versions in `.mtb` files |

### Device Configurator Issues

| Symptom | Cause | Fix |
|---|---|---|
| Changes in Device Configurator don't take effect | Forgot to save, or editing wrong BSP | Verify you're editing the **active** BSP's `design.modus`; save (`Ctrl+S`) |
| `GeneratedSource/` files look wrong or outdated | Stale generated files | Re-save `design.modus` in Device Configurator to force regeneration |
| `cybsp.h` macros don't match pin config | `cybsp.h` not updated after Device Configurator change | Manually update `cybsp.h` macros to match new pin assignments |

### General Tips

- **When in doubt, clean and rebuild:** `make clean && make getlibs && make build`
- **Check `libs/mtb.mk` exists:** If it's missing, `make getlibs` hasn't been run.
- **Verify active BSP:** Open `Makefile` and check the `TARGET=` line — it should match your intended BSP directory name (without the `TARGET_` prefix in the Makefile, but WITH `TARGET_` in the `bsps/` directory name).
- **Read the library README:** If a newly added library doesn't work, check `mtb_shared/<library>/<version>/README.md` for required `COMPONENTS` and `DEFINES`.

## Appendix C: Key Concepts Reference

### The ModusToolbox Project Structure

```
my-project/
├── Makefile                    ← Project build configuration
├── main.c                      ← Your application code
├── deps/                       ← YOUR declared dependencies (committed to Git)
│   └── retarget-io.mtb
├── bsps/                       ← BSP files (local to project)
│   └── TARGET_APP_KIT_PSC3M5_EVK/
│       ├── cybsp.c / cybsp.h   ← Board init API
│       └── config/
│           └── design.modus    ← Device Configurator master file
├── libs/                       ← Resolved dependency tree (generated, never commit)
│   ├── mtb.mk                  ← Generated SEARCH paths
│   └── (transitive .mtb files)
├── build/                      ← Build output (never commit)
└── mtb_shared/                 ← SHARED library source (one level up, never commit)
    ├── mtb-hal-psc3/
    │   └── release-v1.1.2/
    ├── retarget-io/
    │   └── release-v1.9.0/
    └── ...
```

### `deps/` vs `libs/` vs `mtb_shared/` — Know the Difference

| | `deps/` + `bsps/` | `libs/` | `mtb_shared/` |
|---|---|---|---|
| **What's inside** | `.mtb` files you declared (direct dependencies + BSP) | Resolved transitive `.mtb` files + generated `mtb.mk` build wiring | Actual cloned library source code |
| **Who controls it** | You (the developer) | `make getlibs` (the tool) | `make getlibs` (the tool) |
| **Commit to Git?** | ✅ Yes — these are your declarations | ❌ No — reconstructed by `make getlibs` | ❌ No — reconstructed by `make getlibs` |
| **Scope** | Per-project | Per-project | Per-workspace (shared across projects) |

### What to Commit, What to Ignore

| ✅ COMMIT | ❌ IGNORE |
|---|---|
| `Makefile` | `build/` |
| `main.c` and all source files | `mtb_shared/` |
| `deps/*.mtb` | `libs/` |
| `bsps/` (BSP customizations) | `.vscode/` |
| `.gitignore` | `.cache/` |
